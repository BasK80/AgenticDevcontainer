Why sandbox an agent

An unsandboxed agent has
your entire machine’s blast radius

An AI agent runs real commands and makes real network calls on partly-untrusted input. Without a boundary, a single bad decision — or a prompt injection — reaches everything you can.

⚠ Credential theft

API keys, cloud tokens, ~/.ssh, and .env files are all readable — and quietly exfiltrated over any open egress.

⚠ Host compromise

A mounted Docker socket or root access turns a wrong command into a one-line takeover of the host machine.

⚠ Unbounded egress

Free outbound network is an open channel to pull in malware or push your code and secrets anywhere.

The goal: blast-radius containment. Limit what a running agent can touch to the project workspace and an explicit allowlist of network destinations — no host home directory, no cloud credentials, no SSH keys, no cross-project bleed.

Default-deny egress

Everything blocked — except an
explicit, auditable allowlist

  • ▸Domain-based, not IP. The allowlist is hostnames — resilient to CDN / Azure IP rotation, no re-resolution needed.
  • ▸Baseline + feature-sets. A tiny always-on baseline plus toggleable sets (anthropic, github, npm, azure, pypi…). Turn off what you don’t use to shrink the exfil surface.
  • ▸Readable 403, not a cryptic failure. A blocked request returns a plain-text page explaining the host is off the allowlist — so agents surface actionable guidance instead of retrying.
  • ▸Hot-reload & live dashboard. Changes take effect in ~5s; a localhost-only dashboard streams every request, green (allowed) / red (denied).
  • ▸Long-term audit log. Every request — allowed and denied — is persisted to a SQLite log (retention configurable, default 2 months). Query it by time / host / decision via fw audit or the dashboard, and export any period to CSV.

Defense in depth

The rest of the perimeter

⚿ Non-root, no sudo

Runs as devuser (UID 1000) with no privilege-escalation path whatsoever.

⚿ No Docker socket

/var/run/docker.sock is never mounted — mounting it is a one-line host root escalation.

⚿ Resource caps

CPU (4 cores), memory (8 GB), PID (512) limits stop a runaway agent from hurting the host.

⚿ Project-scoped volumes

All state lives in per-project named volumes. Separate projects = separate state. No bleed.

⚿ Read-only git identity

~/.gitconfig is mounted read-only; SSH keys stay on the host (sign-only, opt-in).

⚿ Extensions are a trust boundary

A host-controlled, read-only pinned set — an in-container agent can’t add to it.

Validate it, don’t trust it. The bundled /security-test skill runs escape / exfiltration / tamper probes from inside the container and reports per test: HELD (blocked = pass) or BYPASS (a finding). Re-run after any config change.

Pluggable LLM providers

One switch, many backends

A single use-* command re-points both claude and opencode at a different provider and keeps them in sync. The choice persists across restarts and rebuilds.

use-anthropic Anthropic subscription (OAuth) — the default. No static key on disk; the leaked key is actively unset in each new shell. ★ recommended
use-foundry Azure AI Foundry (Entra ID) — the keyless enterprise alternative, browser-login flow supported in-container.
use-anthropic-key Anthropic API key / gateway — the static-credential fallback, selected explicitly when you need it.
copilot GitHub Copilot — Copilot-backed models via opencode and the Copilot CLI (enable the copilot feature-set).
OAuth / Entra first, static keys deprioritised. Identity-based auth means there’s no long-lived secret sitting in the workspace for an agent to read — and the firewall narrows the exfil surface even when a key is in use.
